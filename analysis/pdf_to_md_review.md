# PDF → Markdown 转换工程分析审计

## 1. 当前实现概览

- 入口：`file_ingest::to_markdown` -> `renderer::file_to_markdown` -> `DocumentPipeline::ingest_bytes` -> `parser::parse` 分发到 `parser/pdf.rs`。
- PDF 文本提取：仅使用 `pdf_extract::extract_text(&result.path)`。无版面分析、无布局重建、无分页语义保留。
- 文本预处理：仅做中日韩字符间空格的归一化（`normalize_cjk_spacing`），不做段落/标题/列表的结构推断。
- AST 构建：`DocumentPipeline::build_document` 先写一个固定 H1 (`"PDF Document"`)，再把文本拆为段落/标题块。标题识别只基于规则：行长<=80、含字母、且全大写比例高或以冒号结尾。
- 段落合并逻辑：逐行读取文本，空行切段；否则行间用空格拼接为段落。
- Markdown 渲染：AST 顺序直接渲染为 Markdown；表格与图片来自其他解析器，但 PDF 解析并未生成表格/图片数据。

## 2. 满足需求的地方

- 有统一 AST 与 Markdown 渲染层，保证输出格式统一（canonical Markdown 的单一路径明确）。
- 输出包含 frontmatter（源路径、类型、checksum、metadata），对后续追踪与可溯源性有帮助。
- 对 PDF 文本做了最基础的 CJK 空格归一化，避免了严重的“中文字符间空格”噪声。

## 3. 存在的主要风险

- **结构保真度严重不足**：
  - 标题层级基本不可恢复，当前标题识别规则偏向英文全大写行，对中文标题几乎无效。
  - 段落合并依赖空行，但 `pdf_extract` 输出常无真实空行，导致章节、段落被错误合并。
  - 列表/表格/代码块无法识别；PDF 的表格与列表会退化成连续文本。
- **噪声控制不足**：
  - 页眉、页脚、页码、水印没有任何过滤策略，且会直接进入正文段落。
  - 版面分栏/页内注释/页码混入正文，易破坏语义连续性。
- **分页与版面信息完全丢失**：
  - 无页边界标记与行位置信息，无法分辨跨页断行或栏目切换。
- **图片内容不可用**：
  - PDF 解析未抽取图片为 `ImageBlock`，图片中的信息无法进入 Markdown；对含图 PDF 直接丢失内容。
- **`preserve_formatting` 选项无效**：
  - Options 提供了 `preserve_formatting`，但 PDF 流程未使用该选项。

## 4. 对“数据清理 + 切片阶段”的潜在影响

- **切片边界不稳定**：段落合并逻辑会把多个独立语义段合并成一段，导致 fragment 过长且边界不可信。
- **标题驱动的切片不可行**：中文标题无法识别，结构化切片（按章/节）失败，后续只能依赖内容长度或规则硬切。
- **噪声传播**：页眉/页脚混入正文，会污染摘要、搜索索引和问答结果。
- **语义稀释**：列表、表格退化成普通文本，信息密度下降，后续清理难以恢复原结构。
- **不可重建版面**：未保留页码、位置信息、字体或块级结构，后续清洗无法“回补”语义段落或标题层级。

## 5. 是否可以作为当前阶段的合格输入（明确判断）

**结论：不合格。**

理由：PDF → MD 当前仅提供“可读文本串”，缺少可靠的标题层级、段落边界与结构化块（列表/表格/图片）识别。对于以 Markdown 为“唯一数据工程入口”的系统，这种输出会导致后续数据清理与切片阶段的结构性失真与噪声扩散，无法满足高质量 canonical Markdown 的要求。
